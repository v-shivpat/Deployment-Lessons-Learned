Deployment Lessons Learned Application

Operations Guide • Internal Documentation

1. Application Overview and Purpose

The Deployment Lessons Learned (DLL) application is an operational workflow system used to systematically capture, assign, track, and resolve deployment issues across data center operations.

This is not a reporting tool first—it is a mechanism to ensure accountability, corrective action, and operational learning across deployments.

  • Standardize issue capture across sites and programs
  • Ensure named ownership using email-based identity
  • Drive corrective and preventative actions (CAPA)
  • Reduce repeat deployment failures
  • Provide leadership visibility into execution risk
If a deployment issue is not captured in this system, it will likely happen again.

3. Submitting a Lesson Learned (Intake Form)

When to Submit

Condition Guidance
Deployment impact Submit when an issue impacts schedule, cost, safety, or quality.
Rework or workaround Any issue requiring rework, workaround, or deviation should be captured.
Future deployment risk Risks that could affect future deployments belong here.
Escalation required If time, money, or escalation was involved, a lesson must be submitted.

How to Submit

Use the Intake Form to capture the lesson. The quality of this entry directly impacts how quickly the issue can be resolved and acted upon.

4. End-to-End Lesson Lifecycle

End-to-End Lesson Lifecycle

From intake → ownership → CAPA → execution → closure

  1. 1

    Lesson submitted via Intake Form

    Entry created in Dataverse

  2. 2

    Status set to New

    Ready for triage

  3. 3

    AMER team reviews and assigns ownership

    Owner team + assignee identified

  4. 4

    Additional details completed

    Improves trend visibility + routing accuracy

  5. 5

    CAPA determination made

    Corrective / preventative action decision

  6. 6

    Actions created if required

    One lesson → many actions

  7. 7

    Status progresses to In Progress

    Execution underway

  8. 8

    Blocked if dependencies exist

    High-signal intervention state

  9. 9

    All actions completed

    CAPA execution finished

  10. 10

    Lesson marked Completed

    Closed with accountability + learning captured

5. Action (CAPA) Management

Action Types

Action Type Purpose
Corrective Action Fixes an identified issue that has already occurred.
Preventative Action Prevents similar issues from occurring in future deployments.

When an Action Is Required

Condition Guidance
Issue requires a fix Create a corrective action.
Risk of recurrence exists Create a preventative action.
No fix or prevention needed No CAPA required; lesson may be closed once documented.
CAPA is not technically required for every lesson. Some lessons exist to document context, inform future decisions, or capture knowledge without generating action items.

Required Action Fields

Field Description
Action Title and Description Clearly defines the work to be performed.
Action Type Identifies whether the action is corrective or preventative.
Due Date Target completion date for the action.
Assigned To Individual accountable for executing the action.
Linked Lesson Parent lesson that the action supports.
Status / State Tracks execution progress (New, In Progress, Blocked, Completed, Closed).

Optional Action Fields

Field When to Use
Priority Used to indicate urgency relative to other actions.
Linked Lesson Owner Provides visibility for coordination and escalation.

Each action is individually owned, monitored, reminded, and escalated independently, even when multiple actions are associated with the same lesson.

6. Automation and Notifications

Design Principle

Principle Meaning
Notify on responsibility Emails are sent only when the recipient has a clear action to take.
No notification noise Status changes without required action do not generate emails.

Lesson Submission Notification Scenarios

Scenario Trigger Condition Notifications Sent
Scenario A – Different people Created By ≠ Assigned To Confirmation email to submitter; action request email to assignee.
Scenario B – Same person Created By = Assigned To Single combined confirmation + action email.
Scenario C – Unassigned Assigned To is blank Email to submitter requesting details completion or assignment.

Details Completion Reminders

Timing Condition Recipient Outcome
Day 5 Details incomplete Assignee (or submitter if unassigned) Reminder email with missing fields listed.
Day 7 Details still incomplete Contributors group Escalation requesting team assistance.

Action (CAPA) Notification Logic

Event Condition Notification
Action created (same person) Creator = Assignee Single email confirming creation and assignment.
Action created (different people) Creator ≠ Assignee Confirmation to creator; assignment email to assignee.
Action reassigned Assigned To updated Immediate notification to new assignee.

Status‑Based Alerts

Transition Recipients Reason
Lesson → Blocked Assignee + Contributors High‑signal state requiring intervention.
Lesson → Completed / Closed Submitter + Contributors Closes the loop for stakeholders.
Action → Blocked Assignee + Lesson Owner CAPA cannot progress without attention.
Action → Completed / Closed Creator + Lesson Owner + Contributors Confirms CAPA resolution.

Due‑Date Reminder Cadence

Timing Recipients Purpose
T‑2 days before due date Action assignee Advance notice.
On due date Action assignee Due‑today reminder.
Every 5 days overdue (max 3) Assignee, then lesson owner Escalating reminders.
After 3 reminders Contributors group Manual team escalation.

Weekly Contributors Digest

Schedule Recipients Contents
Every Monday, 8:00 AM Contributors group Summary of open lessons, incomplete details, blocked items, overdue actions, and closures for the prior week.

Details Completion Gate (IsDetailsComplete)

Concept Definition
IsDetailsComplete Calculated flag indicating whether sufficient additional lesson details have been completed.
Threshold logic TRUE when a minimum number of key fields are populated (recommended: 5 of 10).
Purpose Controls Day‑5 reminder and Day‑7 escalation automation.

Extended Lesson Aging Escalations

Condition Recipient Outcome
Past due date, not closed Lesson assignee Due‑date reminder sent.
IsDetailsComplete = FALSE after 14 days Contributors group Flag for team assistance.
No status change for 30+ days Contributors group Highlighted in weekly digest for intervention.

Power Automate Flow Architecture

Flow Trigger Responsibility
Lesson Created Record created Scenario A/B/C branching and Day‑0 emails.
Details Reminder Scheduled (daily) Day 5 reminders and Day 7 escalation.
Lesson Updated Record updated Status and reassignment alerts.
Action Created Record created Action assignment notifications.
Action Updated Record updated Blocked, completed, and closed alerts.
Overdue Reminders Scheduled (daily) Due‑date and escalation cadence.
Weekly Digest Scheduled (weekly) Executive‑level operational summary.

7. Roles and Permissions

Role Responsibilities
